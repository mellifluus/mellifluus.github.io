1. Introduction

This document is meant to illustrate how various algorithms work and the data structures that these make use of. Starting from a theoretical refresh on some important mathematical concepts (widely used in the science of analyzing algorithms), I’m going to use these concepts to analyze how complex various data structures and algorithms are. I’m also going to go over pros and cons on each and every one of them, giving an idea on which use cases suit them best.

Needless to say, this document requires basic understanding of programming (going to demonstrate concepts with Python in some cases) and a basic mathematical knowledge.

2. Algorithms Analysis Theory

In this section, I’m going to give an overview on the practices used to analyze algorithms. It’s imprecise to just measure how much time or space an algorithms requires to run to determine its complexity. That would just give an information on how much resources the algorithm takes to execute on one specific machine, but doesn’t really give unbiased information on what’s the amount of resources required on a worse or even better machine. That is why, while analyzing algorithms, we use specific notation that allows for clearer understanding of the complexity of an algorithm.

2.1. Complexities

There are two types of complexities to take into consideration when designing an algorithm:

  • Time complexity – Attempts to give a broad idea of how complex a given algorithm is in correlation with its execution time.
  • Space complexity – Attempts to give a broad idea of how complex a given algorithm is in correlation with the space required for its execution. It consists of two parts, which are summed together:
    • Auxiliary space – Represents the amount of space used by variables and constants that are not dependent by the size of the input of the algorithm. In other words, the temporary space required.
    • Variable space – Represents the amount of space used by the input size of the algorithm.

Both complexities can be expressed using Asymptotic Notation, which is going to be covered in depth in the next chapter. In this document, I’m going to be – almost – solely focus on the concept of time complexity.

2.2. Asymptotic Bounds

Any given algorithm takes an n amount of steps to complete. As stated before, we can’t just measure the time a step takes to complete as that wouldn’t give us a realistic complexity representation. That is why, to express the complexity of an algorithm we make use of a concept called Asymptotic Notation, which allows for a machine-independent measurement.

The notation makes use of the concept of Asymptotic Bound which is, by definition, a curve representing the limit of a function where the distance between the function and the curve tends to zero.

When analyzing an algorithms, we’re interested in three types of bounds (each with its own notation).

2.2.1. Asymptotic Upper Bound – \(Big\mbox{-}O\)

The upper bound, which can also be referred to as \(Big\mbox{-}O\) or just \(O\) represents the worst-case scenario in the running time of an algorithm. Let’s say \(f(n)\) describes the running time of an algorithm; we can then say that \(f(n)\) is \(O(g(n))\) if and only if a constant \(c\) exists such that \(c \cdot g(n) \geq f(n)\) for all \(n \geq n_0\).

In mathematical terms, we can then define \(Big\mbox{-}O\) as follows:

\(f(n) = O(g(n)) \iff c \cdot g(n) \geq f(n) \;\; \forall \;\; n \geq n_0 \)

More than often, there is more than one function that is suitable as \(Big\mbox{-}O\) for one algorithm, but we’re interested in the tightest (closer) upper bounding function among them, since it gives us a more realistic idea of the worst-case running time of a function.

fig. 1 – from Programiz

2.2.2. Asymptotic Lower Bound – \(Big\mbox{-}\Omega\)

The lower bound, which can also be referred to as \(Big\mbox{-}\Omega\) or just \(\Omega\) represents the best-case scenario in the running time of an algorithm. Let’s say \(f(n)\) describes the running time of an algorithm; we can then say that \(f(n)\) is \(\Omega(g(n))\) if and only if a constant \(c\) exists such that \(c \cdot g(n) \leq f(n)\) for all \(n \geq n_0\).

In mathematical terms, we can then define \(Big\mbox{-}\Omega\) as follows:

\(f(n) = \Omega(g(n)) \iff c \cdot g(n) \leq f(n) \;\; \forall \;\; n \geq n_0 \)

Again, there is usually more than one function that is suitable as \(Big\mbox{-}\Omega\) for one algorithm, but we’re interested in the tightest (closer) lower bounding function among them, since it gives us a more realistic idea of the best-case running time of a function.

fig. 2 – from Programiz

2.2.3. Asymptotic Tight Bound – \(Big\mbox{-}\Theta\)

The tight bound, which can also be referred to as \(Big\mbox{-}\Theta\) or just \(\Theta\) represents the average-case scenario in the running time of an algorithm. Let’s say \(f(n)\) describes the running time of an algorithm; we can then say that \(f(n)\) is \(\Theta(g(n))\) if and only if two constants \(c_1, c_2\) exist such that \(c_1 \cdot g(n) \leq f(n) \leq c_2 \cdot g(n)\) for all \(n \geq n_0\).

In mathematical terms, we can then define \(Big\mbox{-}\Theta\) as follows:

\(f(n) = \Theta(g(n)) \iff c_1 \cdot g(n) \leq f(n) \) \( \leq c_2 \cdot g(n) \;\; \forall \;\; n \geq n_0\)

Naively, the asymptotic tight bound is a function which, when multiplied by two different constants, manages to “sandwich” \(f(n)\).

3. Analysis of Data Structures

Data structures are the foundation of most algorithms. They provide a way to store, organize and retrieve data, and each of them has different functionalities, advantages and disadvantages. Knowing how common data structures work and how complex it is to do various operations on them is extremely important for algorithm design – thus I'm going to cover them before the Algorithm Analysis section.

Data structures can be classified in primitive and non-primitive structures:

  • Primitive Data Structures – They're the building block on which non-primitive data structures a built on. They hold one value (i.e. Integer, Float, Boolean...) in a specific location in memory.
  • Non-primitive Data Structures – They're based on primitive data structures, and can be seen as collections of them. They classify in contiguous and linked structures:
    • Contiguous Data Structures – They store data in contiguous (adjacent) locations in memory. The most known example is the Array data structure.
    • Linked Data Structures – They store data in random locations in memory. Each memory cell stores a value and one or more memory pointers to other elements. The most known example is the Linked List data structure.

3.1. Array

Arrays are contiguous data structures, meaning that the data they hold is located in contiguous memory locations. This makes accessing the elements in the array efficient, since each one of them is assigned an index (fig.5). Two dimensional arrays (fig. 6) can be used to represent matrixes in code by having an array which has other arrays as its elements.

Arrays have different properties depending on the programming language implementation. In C, for example, arrays have a fixed size and only allow elements of one data type. In Javascript instead, their size is dynamic (their size doubles every time they're full) and the developer is allowed to store whatever type they prefer into them.

Arrays are an extremely common data structure, and other structures, that we're going to look into later are based on them.
Space complexity-wise, they're extremely efficient since they contain only data at every index.

3.2. Linked List

Linked Lists are non-linear data structures which means that, even though they're a collection of data, they don't store it in contiguous memory locations. They instead make use of pointers to keep track of every element in the list, which are themselves stored as part of each element.
The first element of the list is the "HEAD" of the list, and that's the only element that the structure itself needs to keep a reference of, since all subsequent elements are keeping references of each other.

    • Circular singly linked list (fig. 9) – It's a singly linked list where the last node points back at the first node (HEAD node)
    • Circular doubly linked list (fig. 10) – It's a doubly linked list where the last node points back at the first node (HEAD node), and the first node points to the last node in the chain.

3.3. Stack

      

        

          
A stack can be visualized like a pile of heavy books. I can add one more book to it, I can take one book from it, but I can't take the books which are lower in the pile until I relieve them of the weight on them. In computer science, this principle is called L.I.F.O. (Last In, First Out).

          
Stacks can be implemented using either an array or a linked list and, in general, the best advantage of implementing stacks with the latter data structure is their dynamic size. There's no other big difference between the implementations since the stack only needs to keep a reference of the top element.

3.4. Queue

      

        

          
A queue can be visualized like a line of people waiting for their turn at the post office. More people can join the queue by standing at the end of the line, but they'll have to wait for their turn because the post office only serves the person in front of the line, one at a time. In computer science, this principle is called F.I.F.O. (First In, First Out).

          
Queues can be implemented using either an array or a linked list and, in general, the latter implementation's best advantage is its dynamic size. In addition to that, the linked list implementation doesn't run into the problem of running out of space after consecutive successful dequeues – something that gets addressed in array implementations using circular queues.

3.6. Tree

      

        

          
Trees are non-linear data structures that consist of nodes and edges. Every node consists of data and pointers to other nodes. The top node is also called root node and has no parent node. A node is a leaf or external node when it doesn't point to any other node – they're usually located towards the bottom of the tree if the tree is balanced. Edges are basically the connections between nodes, and there is a maximum of \(n - 1\) edges in a tree that has \(n\) nodes.

          
Since trees can be implemented in many different ways, there are some properties that describe them. I'll take into consideration fig. 17 while defining them: 

          
    
            
  • Degree: The maximum number of children nodes allowed for every node. In this case, \(3\).
    • 
            
  • Depth: The number of edges between a node and the root node. In this case, depth is \(0\) for the root node, \(2\) for the rightmost node and \(3\) for the leftmost one.
    • 
            
  • Height: Starting from \(0\) for the leaf nodes, it defines the distance between external nodes and the root node. The root node, in this case, has a height of \(3\) so the whole tree also has a height of \(3\).
    • 
            
  • Level: Starting from \(1\) for the root node, it defines the distance between a node and the root node. In this case, level is \(1\) for the root node, \(3\) for the rightmost node and \(4\) for the leftmost one.
    • 
          

          
As stated before, there are many ways of implementing a tree. In a Skewed binary tree for example, every node has only either a left or right child node, which leads to a structure that is more similar to a linked list.
This document is going to cover only trees of degree \(2\).

3.6.2. Red-Black Binary Tree

      

        

          
Red-black binary trees add a new "color" property to their nodes, which can be either red or black. They implement some more rules on the nodes behaviour based on their color:

          
    
            
  • The root of the tree is a black node.
    • 
            
  • A node cannot have a parent or children node of its same color, except when its child is a leaf node.
    • 
            
  • Leaf nodes are considered black nodes and have a value of null.
    • 
            
  • The path from any node in the tree to any of its leaf descendants contains the same number of black nodes.
    • 
          

          
By implementing these rules, the tree balances itself when inserting or removing nodes.

3.6.3. AVL Binary Tree

      

        

          
An AVL tree (named after its inventors – Adelson-Velsky,Landis) is a self-balancing tree that restores balance if needed after every insertion or deletion. It does that by checking that the height of every node's subtrees does not exceed a threshold, which is usually \(1\). In mathematical terms, an AVL tree has to satisfy \(|B(N)| \leq 1\) for every given node \(N\), where \(B(N) = h_l - h_r\) with \(h_l, h_r\) being the height of the left and right child subtrees respectively. 

          
When \(B(N) > 1\), the tree is left-heavy, and requires either a right rotation or a left-right rotation.

          
When \(B(N) < -1\), the tree is right-heavy, and requires either a left rotation or a right-left rotation.

3.7. Heap

      

        

          
Heaps are data structures that are commonly used in Heap Sort and to implement Priority queues. They can be implemented in two different ways:

          
    
            
  • Min Heap: It's a tree where every node holds a value that is less or equal than its children's held values. The smallest value in the min heap is located at the root of the tree.
    • 
            
  • Max Heap: It's a tree where every node holds a value that is greater or equal than its children's held values. The largest value in the max heap is located at the root of the tree.
    • 
          

          
Heaps are complete trees. That means that a heap with \(n\) nodes always has a \(O(\log(n))\) height. In an array implementation:

          
    
            
  • The root node is stored on index \(1\) and subsequent nodes are stored in order of level traversal from left to right.
    • 
            
  • The element at index \(0\) is often left unused. This is a convention that is not always respected, but it makes the following point more straight-forward.
    • 
            
  • Any node at a given index \(i\) has the following properties:
              
      
                
    • Its left child is located at index \(2i\)
      • 
                
    • Its right child is located at index \(2i + 1\)
      • 
                
    • Its parent is located at index \(\lfloor i / 2\rfloor\)
      • 
              
    
            
    • 
          

        

        

          
          
fig. 22 - Min Heap

        

        

          
          
fig. 23 - Max Heap

        

      

      

        
The process of "sorting" an unordered array to follow heap's rules is usually referred to as "building" the heap. It's a recursive process that starts from the bottom of the tree (\(h=0\)) and works its way to the top. For every subtree iteration, the root value is compared to its children values and bubbled down if needed.

3.8. Hash Table

      

        

          
A Hash Table is a key-value pair based data structure which uses hash functions that return an integer \(i\) which is then used to store the data in an array at index \(i\). Hash functions work by doing operations on the key value which are not reversible (i.e. impossible to derive the key from the hash).

        

        

          
          
fig. 24 - Hash table

      

      

        
Sometimes, the hash function generates an index that is already occupied by another key-value pair. That is called collision, and there's various ways to address the issue:

        
    
          
  • Linear/quadratic probing: When a collision occurs, linear probing searches for the first available empty index. Quadratic probing does the same but "skips" indexes by incrementing an integer quadratically until there's no collision.
    • 
          
  • Double hashing: Generates a new hash when a collision occurs. More efficient than Linear/quadratic probing but inconsistent.
    • 
          
  • Chaining: Chaining stores the new key-value pair on the index even if there is a collision. The pairs are stored in a linked list.
    •

4. Analysis of Algorithms

        
Many aspects of our digital lives are controlled by algorithms – from the search function of a music streaming service to a navigation app that guides us to our destination.

        
Because there's so many possible applications for algorithms in everyday life, various design techniques distinguish the multitude of algorithms ever created: 

        
    
          
  • Brute force algorithms: There's no real technique involved here – these algorithms just try every possible combination and stop executing when they find a solution. Linear search is a good example.
    • 
          
  • Greedy algorithms: They're algorithms that always follow the optimal path, without regards of the efficiency of the final solution.  Dijkstra's algorithm is an example of greedy algorithm.
    • 
          
  • Backtracking algorithms: Backtracking algorithms are usually recursive algorithms that try to work towards a solution by combining multiple steps. If a part of the solution doesn't lead to a final solution, it is removed and another solution is looked for. They're usually implemented in combination of dynamic programming, which is a technique that stores the results of a function with various inputs, so that, if the function is called with the same inputs again, the result is already stored and there is no need to run the function again. Rat in a Maze Algorithm is a good example.
    • 
          
  • Divide and conquer algorithms: These algorithms try to divide the problem in smaller sub-problems to solve those instead. In some algorithms, the sub-problems solutions are sometimes recombined in the final solution. Merge sort is a classic divide and conquer algorithm.
    • 
        

      

    

    

      
4.1. Master Theorem

      

        

          
Lastly, before diving into Algorithm Analysis, it is worth mentioning that many divide and conquer algorithm's time complexity can be calculated using the Master Theorem. It is a formula that helps solving recurrence relations of the form:

          

            $$
            \begin{aligned}
            T(n) &= aT(\frac{n}{b}) + f(n) \;\; \forall \;\; a \geq 1,\; b > 1,\; f(n) \geq 0
            \end{aligned}
            $$
          

          
Where \(T(n)\) is the time complexity of the algorithm, \(a\) is the number of sub-problems, \(T(\frac{n}{b})\) is the runtime of each initial subproblem and \(f(n)\) is the time complexity of the algorithm's work done outside of the recursive calls.

        

        

          
          
fig. 25 - Master theorem tree

        

      
  
      

        
At depth \(d\), a complete tree has \(a^d\) nodes and the maximum depth of a recursion tree is \(\log_b(n)\).
That means that to find out how many leaves the tree has we can calculate \(a^{\log_b(n)}\) which, for logarithms properties equals to \(n^{\log_b(a)}\).

        
A running time of \(\Theta(n^{\log_b a})\) follows.

        
Since in the master theorem – \(T(n) = aT(\frac{n}{b}) + f(n)\) – we're adding \(f(n)\) to the recursive calls complexity, the next step is comparing the two values and finding out which one dominates. There is three cases:

        

          $$
          \begin{aligned}
          &\text{Case 1: if} \; f(n) = O(n^{\log_b(a) - \epsilon}) \;\; \forall \;\; \epsilon > 0 \text{, then} \; T(n) = \Theta(n^{\log_b(a)})\\
          &\text{Case 2: if} \; f(n) = \Theta(n^{\log_b(a)}) \text{, then} \; T(n) = \Theta(n^{\log_b(a)}\log(n))\\
          &\text{Case 3: if} \; f(n) = \Omega(n^{\log_b(a) + \epsilon}) \;\; \forall \;\; \epsilon > 0 \text{, then} \; T(n) = \Theta(f(n))
          \end{aligned}
          $$     
        

      

    

    

      
4.2. Search Algorithms

      
Search algorithms are algorithms that try to find a specific element in a data structure. There are many different types of search algorithms, each with their own advantages and disadvantages.

    

    

      
4.2.1. Linear Search

      
Linear search is a very simple search algorithm that works by iterating through the data structure and comparing each element with the element we're looking for. If the element is found, the algorithm returns the index of the element. If the element is not found, the algorithm returns -1.

      
Linear search is a brute force algorithm, which means that it tries every possible combination and stops executing when it finds a solution.

4.2.2. Jump Search

      
Jump search is a search algorithm for sorted arrays. It works by jumping a fixed number of steps at a time, and then doing a linear search on the elements in that block – in case the jump finds an element greater than the searched element. If the element is found, the algorithm returns the index of the element. If the element is not found, the algorithm returns -1.

      
Jump search is a divide and conquer algorithm, which means that it divides the problem in smaller sub-problems to solve those instead. In this case, the sub-problems are the blocks of the array.

      
The optimal step size is \(s=\sqrt{n}\), since in the worst case \(\frac{n}{s}\) steps are made, with the addition of \(s-1\) steps for the linear search. \(\sqrt{n}\) provides the minimum amount of steps for \(\frac{n}{s} + (m - 1)\).

4.2.3. Binary Search

      

        

          
Binary search is a search algorithm for sorted arrays. It works by dividing the array in half and checking if the element we're looking for is greater or lesser than the current element. The algorithms then proceeds to only look in one half of the array, dividing that in two again and so on. If the element is found, the algorithm returns the index of the element. If the element is not found, the algorithm returns -1.

          
Binary search is a divide and conquer algorithm, which means that it divides the problem in smaller sub-problems to solve those instead. In this case, the sub-problems are the halves of the array.

          
To calculate the complexity of the recursive implementation of the binary search algorithm, we can make use of the Master Theorem. We just need to plug these values in it:

          
    
            
  • \(a=1\) - since there's only one sub-problem for every recursion.
    • 
            
  • \(b=2\) - since the problem space is halved for every recursion.
    • 
            
  • \(f(n)=O(1)\) - since the work done outside of recursive calls is upper bounded by \(O(1)\).

4.4. Complex Algorithms

      
Until now, we analyzed common algorithms. In general, sorting and searching arrays is often considered trivial. We're now going to go over some more complex algorithms. Some of these have more concrete applications in real life.

    

    

      
4.4.1. Rat in a maze algorithm

      

        

          
The rat in a maze algorithm is a backtracking algorithm that solves the following problem:

          
Given a maze, represented by a 2D array, find a path from the top left corner to the bottom right corner. The maze is surrounded by walls, and the rat can move in all four directions – up, down, left and right.

          
The algorithm works by recursively trying to move the rat in all directions. If the rat can't move in any direction, it backtracks and tries another path.

          
This algorithms is a good example of a backtracking algorithm.

          
In this implementation of the algorithm, the matrix that holds the maze has the same width and height, so the total number of cells in the maze is \(n^2\). For every cell, there is four possibilities of movement – also called unvisited neighbours.

          
Since the algorithm tries every possible path, the time complexity is \(O(4^{n^2})\).

            \(\Theta(4^{n^2})\)
            The average case is the same as the worst case.

4.4.2. Dijkstra's Algorithm

      

        
Dijkstra's Algorithm is one of the most famous algorithm ever invented. It takes its name from its inventor, computer scientist Edsger Wybe Dijkstra. The algorithm is used to find the shortest path between two vertices in a weighted graph. It requires the weights on every edge to be positive to work.

        
The algorithm works by maintaining a set of vertices that have been visited, and a set of vertices that haven't been visited. The algorithm starts by adding the source vertex to the set of visited vertices, and all other vertices to the set of unvisited vertices. Then, the algorithm iterates over the unvisited vertices, and finds the vertex with the smallest distance from the source vertex. The algorithm then adds this vertex to the set of visited vertices, and removes it from the set of unvisited vertices. The algorithm then updates the distances of all unvisited vertices that are adjacent to the newly visited vertex. The algorithm repeats this process until all vertices have been visited.

        
The algorithm is a good example of a greedy algorithm. It always chooses the vertex with the smallest distance from the source vertex. This is a good heuristic, but it doesn't always lead to the optimal solution.

      

      

        

        
C++ implementation of Dijkstra's Algorithm - From geeksforgeeks.org.

      

      

        
Dijkstra's algorithm has many steps, so to measure its time complexity we have to take them all into account:

        
    
          
  • Getting a node from the heap has a time complexity of \(O(\log(|V|)\)
    • 
          
  • Comparing the costs of all edges has a time complexity of \(O((|E|)\)
    • 
          
  • The length of the path returned by the algorithm can't be more than \(O(|V|)\)
    • 
        

        
Since the algorithm iterates over all vertices, the time complexity is \(O(|V|\log(|V|) + |E|)\). This is the same as \(O(|E|\log(|V|))\).

      

    

    

      
4.4.3. A* Algorithm

      

        
The A* algorithm is a pathfinding algorithm that is used to find the shortest path between two vertices in a weighted graph. It is a variant of Dijkstra's algorithm, and it is often used in games. The algorithm works by maintaining a set of vertices that have been visited, and a set of vertices that haven't been visited. The algorithm starts by adding the source vertex to the set of visited vertices, and all other vertices to the set of unvisited vertices. Then, the algorithm iterates over the unvisited vertices, and finds the vertex with the smallest distance from the source vertex. The algorithm then adds this vertex to the set of visited vertices, and removes it from the set of unvisited vertices. The algorithm then updates the distances of all unvisited vertices that are adjacent to the newly visited vertex. The algorithm repeats this process until all vertices have been visited.

      

      

        

        
C++ implementation of A* Algorithm - From geeksforgeeks.org.

      

      

        
Instead of just choosing the shortest distance from the source vertex, the A* algorithm also takes into account the distance from the destination vertex. This is done by adding the distance from the source vertex to the destination vertex to the distance from the source vertex to the current vertex.

        
This extremely efficient heuristic function allows the A* algorithm to often have best case scenarios of \(\Omega(e)\) where \(e\) is the number of edges between the start and the destination vertex. The worst case scenario has still a better complexity than Dijkstra, with \(O(|V|+|E|)\).
